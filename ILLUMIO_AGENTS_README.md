# Illumio Suite of Agents

A collection of LangGraph-based AI agents for querying and analyzing Illumio network traffic data stored in Elasticsearch. The suite provides both a unified expert orchestrator and three specialized sub-agents, each focused on a distinct category of network visibility question.

---

## Architecture Overview

```
User Message
    └─> Illumio Expert Agent (orchestrator)
            ├─ classify intent
            ├─> Traffic Agent       — cross-environment flow analysis
            ├─> Blocked Agent       — denied/blocked flow analysis
            ├─> Consumers Agent     — service consumer discovery
            └─> Direct LLM answer  — general Illumio knowledge
```

All agents share the same core pattern:

1. **Parse intent** — LLM extracts structured parameters from free-text input
2. **Build query** — deterministic Elasticsearch DSL construction (no LLM)
3. **Execute search** — query dispatched via MCP `search` tool
4. **Format answer** — human-readable response generated by LLM
5. **Kibana fallback** — if Elasticsearch is unreachable, a ready-to-paste Kibana Dev Tools payload is returned instead

## Agents

### 1. Illumio Expert Agent

**Entry points:** `illumio_expert_agent.py` / `illumio_expert_graph.py`

The recommended single entry point for all Illumio questions. It maintains multi-turn conversation history, remembers entities (application codes, hostnames) across turns, and routes each request to the appropriate specialist agent or answers directly from its built-in domain knowledge.

**Capabilities:**
- Multi-turn conversation with persistent context
- Automatic intent classification (traffic / blocked / consumers / general)
- Entity memory: remembers `ap_code` and `hostname` across turns and enriches sub-agent calls automatically
- Language adaptation: detects the user's language and responds in kind (French or English)
- Graceful error handling: naturalizes both Kibana fallback payloads and error messages via LLM

**Domain knowledge built-in:**
- PCE (Policy Compute Engine) architecture — SNC and MNC topologies
- Label dimensions: App (`A_<AP_CODE>-<name>`), Env (`E_DEV` / `E_REC` / `E_PROD`), Location, Role
- Policy model: Rulesets, enforcement modes, policy decision values
- Elasticsearch traffic flow log field schema

---

### 2. Traffic Agent

**Entry points:** `illumio_agent.py` / `illumio_graph.py`

Answers cross-environment traffic questions such as:
> "Is there traffic flowing between the development and production environments? If so, which applications?"

**Parameters extracted from user input:**
| Parameter | Description | Example |
|-----------|-------------|---------|
| `app_code` | Application code | `AP12345` |
| `direction` | Flow direction | `dev_to_prod` or `prod_to_dev` |

**What it returns:**
- A list of source applications communicating across the requested environment boundary
- Total flow counts per application
- Kibana Dev Tools payload as fallback

**Workflow nodes:**
`parse_intent` → `build_query` → `execute_search` → `format_answer` (or `kibana_fallback`)

---

### 3. Blocked Flows Agent

**Entry points:** `illumio_blocked_agent.py` / `illumio_blocked_graph.py`

Identifies denied network flows targeting or originating from a specific application or host:
> "Were there any blocked flows to or from my application/server?"

**Parameters extracted from user input:**
| Parameter | Description | Example |
|-----------|-------------|---------|
| `target` | Hostname or application code | `myserver01` or `AP12345` |
| `target_type` | Type of identifier | `hostname` or `app` |
| `direction` | Which side to inspect | `inbound`, `outbound`, or `both` |

**What it returns:**
- Inbound blocked flows: top sources, destination ports, protocols
- Outbound blocked flows: top destinations, destination ports, protocols
- Results are presented separately per direction when `both` is requested
- Kibana Dev Tools payload(s) as fallback

**Workflow nodes:**
`parse_intent` → `build_query` → `execute_search` → `format_answer` (or `kibana_fallback`)

---

### 4. Service Consumers Agent

**Entry points:** `illumio_consumers_agent.py` / `illumio_consumers_graph.py`

Discovers which applications are actively connecting to a given service:
> "Which applications are consuming my service?"

**Parameters extracted from user input:**
| Parameter | Description | Example |
|-----------|-------------|---------|
| `app_code` | Application code of the target service | `AP12345` |

**What it returns:**
- Ranked list of source applications (consumer apps) with flow counts
- Only considers flows with `policy_decision = Allowed`
- Kibana Dev Tools payload as fallback

**Workflow nodes:**
`parse_intent` → `build_query` → `execute_search` → `format_answer` (or `kibana_fallback`)

## MCP Integration

All agents query Elasticsearch through an MCP (Model Context Protocol) server defined in `mcp_server.py`. The server exposes a single tool:

```
search(index: str, body: dict) -> dict
```

When the MCP tool is unavailable or Elasticsearch returns an error, every agent falls back to generating a Kibana Dev Tools payload that can be pasted directly into the Kibana console.

## Configuration

All agents are configured via `config.yaml`. The relevant sections are:

### `illumio_agent`
```yaml
illumio_index_pattern: "your-index-*"
source_env_field:      "illumio.source.labels.env"
dest_env_field:        "illumio.destination.labels.env"
source_app_field:      "illumio.source.labels.app"
dest_app_field:        "illumio.destination.labels.app"
dev_env_value:         "E_DEV"
prod_env_value:        "E_PROD"
app_prefix_format:     "A_{app_code}-"
agg_size:              50
```

### `illumio_blocked_agent`
```yaml
illumio_index_pattern:    "your-index-*"
policy_decision_field:    "illumio.policy_decision"
denied_value:             "denied"
source_hostname_field:    "illumio.source.hostname"
dest_hostname_field:      "illumio.destination.hostname"
source_app_field:         "illumio.source.labels.app"
dest_app_field:           "illumio.destination.labels.app"
dest_port_field:          "destination.port"
protocol_field:           "network.protocol"
agg_size:                 20
```

### `illumio_consumers_agent`
```yaml
illumio_index_pattern:    "your-index-*"
source_app_field:         "illumio.source.labels.app"
dest_app_field:           "illumio.destination.labels.app"
policy_decision_field:    "policy_decision"
allowed_value:            "Allowed"
app_prefix_format:        "A_{app_code}-"
agg_size:                 20
```

## LLM Configuration

All agents use the same LLM backend:

- **Model:** `openai/gpt-oss-120b` (served via Groq's OpenAI-compatible API)
- **Temperature:** `0` (deterministic outputs)
- **Base URL:** `https://api.groq.com/openai/v1`

---

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Deterministic query building | Elasticsearch DSL is constructed in pure Python — no LLM involved — ensuring consistent, auditable queries |
| LLM used only for intent extraction and formatting | Reduces latency and prevents query hallucinations |
| Kibana fallback on every agent | Ensures usefulness even without live Elasticsearch access |
| Expert agent as single entry point | Simplifies integration; callers do not need to know which sub-agent to invoke |
| Multi-turn conversation in expert agent | Allows users to ask follow-up questions without repeating context |
